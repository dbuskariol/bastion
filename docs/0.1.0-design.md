# Bastion v0.1.0 — design

> Design rationale for v0.1.0. Produced by a dual-model consensus design process (two independent reasoning models investigated the same brief, cross-pollinated positions, and ran a third blind-spot critique pass) before any code was written, then iteratively refined as the same consensus loop caught load-bearing bugs during stabilisation. Investigation artifacts live in the session workspace; this document is the synthesised, scoped, implemented result.

## What it is

Bastion is a macOS menu-bar SSH connection manager: every host you've ever connected to (or saved manually) listed in one place, one-click connect into your terminal of choice, ControlMaster keepalive enabled by default, and first-class support for SSO/FIDO-fronted SSH bastions.

## High-level architecture

Two Mach-Os:

```
┌──────────────┐  ssh -G / ssh -O check         ┌────────────────────┐
│ BastionMenuBar│ ───────────────────────────▶ │ system /usr/bin/ssh│
│ (SwiftUI)    │ ◀─────────────────────────── │                    │
└──────┬───────┘  effective config, master state└────────────────────┘
       │  AppleScript
       ▼
┌──────────────┐
│  Terminal    │
│  app of      │  ssh -fNM <alias>    ┌────────────────────┐
│  choice      │ ──────────────────▶  │ master daemon       │
│              │                      │ ~/.ssh/sockets/…   │
└──────────────┘                      └────────────────────┘

┌──────────────┐  same engine code path
│ bastion CLI  │ ────────────────────▶ (everything the menu app does)
└──────────────┘
```

The menu app and the CLI both link `BastionCore` and use the same `ConnectionEngine`. There is no privileged helper, no daemon, no LaunchAgent. SSH is a user-space activity and Bastion stays inside that boundary.

## Single managed file with sentinel `Include`

Bastion owns exactly one file in `~/.ssh/`:

- `~/.ssh/config.d/bastion.conf` — generated on every save; first comment line says "do not edit by hand".

And exactly one block in `~/.ssh/config`:

```
# BEGIN BASTION MANAGED — do not edit. Remove the whole block to uninstall Bastion.
Include ~/.ssh/config.d/*.conf
# END BASTION MANAGED
```

Sentinel-guarded so `bastion uninstall` removes it cleanly without disturbing the rest of `~/.ssh/config`. The `Include` directive sits at the top of the file so OpenSSH's first-match-wins evaluation gives Bastion's settings precedence over the user's later `Host *` wildcards.

## Two-pass `ssh -G` validation

Every config write goes through two validation passes:

1. **Isolation pass**: write the rendered config to a tempfile under `$TMPDIR`, then run `ssh -G -F <tempfile> <alias>` for every managed host. Asserts our writer produced something OpenSSH accepts in isolation. Errors here roll back the on-disk file to `bastion.conf.prev` and surface the error in the editor.
2. **Integration pass**: run `ssh -G <alias>` against the user's composed config (no `-F`). Compares the effective value of each option against what Bastion expected. Mismatches indicate the user has a `Host *` wildcard or `Match exec` earlier in their config overriding our settings; surfaced as a yellow warning in the editor, not an error.

Both passes catch entire classes of bugs (malformed output, missing `Include`, conflicting wildcards) before the user discovers them at connect time.

## Shared master across alias/hostname variants

Naïvely, OpenSSH's default `ControlPath ~/.ssh/sockets/%C` hashes the typed hostname (`%h`), so `ssh prodjump` and `ssh prodjump.example.internal` get different `%C` hashes → different sockets → different masters → re-FIDO for every alternate spelling. Bastion solves this with a two-stanza emission per managed host:

```
Host prodjump
    HostName prodjump.example.internal
    User alice
    IdentityFile /Users/alice/.ssh/id_rsa
    IdentitiesOnly yes
    ControlMaster auto
    ControlPath ~/.ssh/sockets/bastion-<12hex>-%p-%r
    ControlPersist 8h
    ProxyJump bastion-jump.example.com

Match host prodjump.example.internal
    User alice
    IdentityFile /Users/alice/.ssh/id_rsa
    IdentitiesOnly yes
    ControlMaster auto
    ControlPath ~/.ssh/sockets/bastion-<12hex>-%p-%r
    ControlPersist 8h
    ProxyJump bastion-jump.example.com
```

The `Match host` block re-asserts the multiplex directives + identity directives so the FQDN-typed form (from `git push`, VS Code Remote-SSH, any other tool) lands on the same socket. The `User` directive lives inside the block, not as a `Match host … user <user>` qualifier — a qualifier silently fails to fire when the local Mac user differs from the configured remote user (empirically verified during rubber-duck pass; covered by tests).

The `-%p-%r` tokens in `ControlPath` are load-bearing security: OpenSSH does not validate `(user, host, port)` against an existing master before mux-attaching, so the socket *name* is the segmentation primitive. Without `-%r`, `ssh otheruser@host` could attach to a master authenticated as a different user.

## FIDO/SSO bootstrap as a first-class case

Many corporate SSH bastions require an interactive WebAuthn / hardware-key challenge that `BatchMode=yes` can't satisfy. Bastion routes that case explicitly:

1. **Pre-flight**: `ssh -G <alias>` resolves the effective config (with a 500ms timeout so `CanonicalizeHostname` / `Match exec` hangs can't lock the UI). If ControlMaster isn't enabled but the host is flagged as needing interactive auth, the editor surfaces a transparent auto-upgrade banner.
2. **Fast path 1**: master already alive → `ssh <alias>` multiplexes through, opens a shell tab.
3. **Fast path 2**: `BatchMode=yes` probe with a 2s hard subprocess-killing timeout. If cached creds (1Password, ssh-agent, passkey) authenticate silently, we mark master ready without a touch.
4. **FIDO bootstrap**: launch `ssh -fNM <alias>` in the user's terminal so they can complete the browser+touch dance in plain sight. Poll `ssh -O check` every second. On success: notification + auto-opens a shell tab. On failure (ssh exited without binding the socket): chip flips to a diagnostic message naming the likely cause (second-hop auth required, target unreachable, etc.) — within ~5s, not the previous 180s blind wait.

The `await master` loop also detects "ssh -fNM exited without binding socket" via per-poll process-aliveness check, so cancelled FIDO touches and second-hop auth failures fail fast.

## AppKit-level menu bar surface

The popover is hosted via `NSStatusItem` + `NSPopover` + `NSHostingController(sizingOptions: [.preferredContentSize])`, not SwiftUI's `MenuBarExtra(.window)`. The macOS 13 implementation of `MenuBarExtra(.window)` has multiple known structural bugs:

- Its private NSPanel does not re-negotiate `intrinsicContentSize` when the body's layout shape changes, so the popover clips dynamic content to whatever size it first observed.
- App-scope `@StateObject` subscriptions re-evaluate the Scene primitive on every `@Published` mutation, detaching the popover's observation chain.

The AppKit-level surface gives Bastion dynamic popover sizing as the user adds / removes / expands hosts (`NSHostingController.preferredContentSize` is wired directly into `NSPopover.contentSize`) and a stable observation chain. Both behaviours are independently verifiable from `log stream --predicate 'subsystem == "com.bastion.menu"'`.

## CLI surface

```sh
bastion list
bastion show <alias>
bastion add <alias> --hostname <h> [--user <u>] [--port <p>] [...]
bastion edit <alias> [...]
bastion remove <alias>
bastion connect <alias>
bastion master start|stop|check <alias>
bastion import --source zsh|bash|fish|known-hosts|ssh-config|all
bastion terminal list|set <id>
bastion config doctor
bastion config sync
bastion config rollback
bastion uninstall
```

Every operation the menu app offers is also scriptable from a shell. `bastion config doctor` includes the shared-master parity check (`ssh -G <alias>` vs `ssh -G <hostname>` for every managed host, asserts identical `controlpath`) so misconfigured wildcards in the user's `~/.ssh/config` surface deterministically rather than as silent connect failures.

## Persistence

- **Registry**: `~/Library/Application Support/Bastion/hosts.json` — atomic write, `.prev` rotation, fsync, round-trip read-back to verify integrity.
- **Preferences**: `~/Library/Application Support/Bastion/preferences.json` — default terminal, notification opt-ins.
- **Status cache**: `~/Library/Application Support/Bastion/status-cache.json` — master uptime tracking across menu-app restarts (uses `stat().st_birthtimespec` on the socket file as the kernel-truth source).
- **Keychain**: SSH private-key passphrases under service name `Bastion SSH passphrase`, account = identity-file path. Stored with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` (no iCloud sync, no access from a sleeping Mac).
- **Generated config**: `~/.ssh/config.d/bastion.conf`, `~/.ssh/config.d/bastion.conf.prev`.

## Sparkle integration

Sparkle 2 with EdDSA signature verification:

- Public key baked into `Info.plist` (`SUPublicEDKey`).
- Appcast URL: `https://github.com/dbuskariol/bastion/releases/latest/download/appcast.xml`.
- Daily background check, user-initiated check via the popover's download icon.
- Local builds (no `SUFeedURL` injected at build time) keep Sparkle inert so dev rebuilds don't surprise the user with update prompts.

## Edge cases worth knowing

- **macOS 13 only**: `BastionMenuBar` targets macOS 13. We deliberately do not use any macOS 14+ APIs (`@Observable`, `.symbolEffect(.replace)`, two-parameter `onChange`, `defaultLaunchBehavior`, etc.) so the app builds and runs on the floor we promise.
- **`Window` scenes auto-open at launch** under macOS 13 SwiftUI when there's no `MenuBarExtra` scene. `AppDelegate.closeAutoOpenedWindows()` runs on the next runloop tick after launch and dismisses any Setup / Host-editor windows the framework auto-opened. User-initiated opens (gear button) and the first-launch onboarding path (800ms-delayed notification) both happen after the close pass and stay open.
- **`Match host` evaluates after `HostName` substitution** but BEFORE `CanonicalizeHostname always`'s second-pass canonicalisation. Bastion documents this as a known limitation: users with `CanonicalizeHostname always` who type a short name that canonicalises into a managed FQDN won't get the shared master automatically. Save-time warning when the scanner sees the pattern.
- **iCloud-synced `~/.ssh`**: detected via host-fingerprint mismatch. Non-blocking warning chip in the header explaining that Bastion only writes to `bastion.conf` and the sentinel block, so cross-Mac conflicts are bounded.

## What was deliberately rejected

- **CLI wrapper for shared-master sharing** (`bastion ssh <alias>`). Defeats the point — users want plain `/usr/bin/ssh` from scripts, git remotes, VS Code Remote-SSH, etc. Config-level mux sharing via the two-stanza `Host` + `Match host` shape solves it for every client.
- **Symlinking `%C` hash variants**. Reproduces OpenSSH's hash internals (version-sensitive) and conflicts with the `-%p-%r` segmentation design.
- **Auto-killing legacy masters during migration**. Respects the user's running sessions; detection-and-prompt via UI chip rather than surprise teardown.
- **App-scope `@StateObject coordinator`**. Subscribes the App's body to every `@Published` write on the coordinator, which on macOS 13 cascades into Scene rebuilds + observation churn. Coordinator lives on the `@NSApplicationDelegateAdaptor`-owned `AppDelegate`; views consume it via `environmentObject` injection from inside the `NSHostingController` tree.
- **iOS-style `@Observable` macro**. macOS 13 doesn't have it; we use `ObservableObject` + `@Published` throughout for consistency.
